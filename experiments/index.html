    Running Experiments¶

    This guide explains how to run and configure experiments with different recommendation algorithms in the QPERA project.

    1. Quick Commands¶

    Full Experiment Suite¶

    To run all predefined experiments across all datasets and scenarios: 

    make run-all

    Viewing Results¶

    All results are tracked with MLflow. Start the UI to view them: 

    make run-mlflow
    Then, navigate to http://127.0.0.1:8080 in your browser.

    2. Experiment Configuration¶

    This project uses a configuration-driven approach. The ExperimentRunner class in qpera/main.py orchestrates all experiments without requiring command-line arguments.

    Predefined Experiment Matrix¶

    The ExperimentRunner._define_experiments() method defines the core combinations of algorithms and datasets: 

    # Located in qpera/main.py → ExperimentRunner._define_experiments()
[
    {"algo": "CBF", "module": CBF, "func": "cbf_experiment_loop", "dataset": "movielens", "rows": 14000},
    {"algo": "CF",  "module": CF,  "func": "cf_experiment_loop",  "dataset": "movielens", "rows": 14000},
    {"algo": "RL",  "module": RL,  "func": "rl_experiment_loop",  "dataset": "movielens", "rows": 14000},
    {"algo": "CBF", "module": CBF, "func": "cbf_experiment_loop", "dataset": "amazonsales"},
    {"algo": "CF",  "module": CF,  "func": "cf_experiment_loop",  "dataset": "amazonsales"},
    {"algo": "RL",  "module": RL,  "func": "rl_experiment_loop",  "dataset": "amazonsales"},
    {"algo": "CBF", "module": CBF, "func": "cbf_experiment_loop", "dataset": "postrecommendations", "rows": 14000},
    {"algo": "CF",  "module": CF,  "func": "cf_experiment_loop",  "dataset": "postrecommendations", "rows": 14000},
    {"algo": "RL",  "module": RL,  "func": "rl_experiment_loop",  "dataset": "postrecommendations", "rows": 14000},
]

    Experiment Scenarios¶

    For each algorithm–dataset combination, the ExperimentRunner._build_configurations() method generates three types of scenarios:

    1. Clean (Baseline): The algorithm runs on the original, unmodified data (limited to a maximum of 14,000 rows).
    2. Privacy (Robustness): Item metadata is progressively hidden to test algorithm robustness. Values in the title and genres columns are replaced with NaN at four levels: 10%, 25%, 50%, and 80%. This simulates privacy-preserving scenarios where user data is incomplete or anonymized.
    3. Personalization: A configurable fraction (10%, 25%, 50%, 80%) of each user's selected items is replaced with items randomly drawn from a probability distribution based on global item frequency. The associated metadata and ratings are updated accordingly to simulate profile dilution with popular items.

    Additional Robustness Scenario

    The thesis also evaluates a second robustness modification where entire interaction rows are removed from the dataset (at the same 10%–80% levels), simulating row-level data loss. This tests the algorithms under conditions of missing interaction records rather than just hidden metadata.

    3. Available Algorithms¶

    Content-Based Filtering (CBF)¶

    • Implementation: A TF-IDF-based content filtering algorithm from the Microsoft Recommenders library.
    • Approach: All descriptive data (title, genres) are merged into a single text field before being tokenized for TF-IDF modeling. Recommendations are generated by comparing a user's past choices with similar items identified through vector similarity.
    • Key Characteristic: Relies on the intrinsic attributes of items, making it sensitive to the quality and completeness of item metadata.

    Collaborative Filtering (CF)¶

    • Implementation: Cornac BPR (Bayesian Personalized Ranking) from the Microsoft Recommenders library.
    • Approach: A pairwise ranking approach that optimizes recommendations by learning to distinguish between preferred and non-preferred items based on implicit user feedback. Operates exclusively on the user-item interaction matrix.
    • Key Characteristic: By design, completely blind to item content metadata — it relies only on user-item interaction patterns.

    Reinforcement Learning (RL)¶

    • Implementation: PGPR (Policy-Guided Path Reasoning) agent that navigates a knowledge graph.
    • Approach: Uses RL to navigate a knowledge graph. The agent starts from a user and learns to find paths to items, where these paths serve as explanations for recommendations.
    • Key Components:
      • Knowledge Graph: Built from users, items, and their attributes (genres).
      • TransE Embeddings: Learns representations for entities and relations.
      • Actor-Critic Agent: Learns a policy for navigating the graph using PPO.
    • Key Characteristic: Capable of generating explainable, path-based recommendations but computationally intensive and sensitive to data quality.

    4. Evaluation Metrics¶

    The project computes a comprehensive set of metrics for k=1 and k=10, all logged to MLflow.

    Accuracy Metrics¶

    • Precision@k: Proportion of users for whom at least one relevant item appears in their top-k recommendations.
    • Recall@k: Average proportion of relevant items successfully included in the top-k list per user.
    • MAE (Mean Absolute Error): Average magnitude of errors between predicted and actual ratings.
    • RMSE (Root Mean Squared Error): Square root of the average squared differences between predicted and actual ratings.

    Ranking Quality Metrics¶

    • NDCG (Normalized Discounted Cumulative Gain): Evaluates ranking quality considering both relevance and position.
    • MRR (Mean Reciprocal Rank): Measures how early the first relevant item appears in the recommendation list.

    Coverage Metrics¶

    • User Coverage: Percentage of users for whom meaningful recommendations (within a rating error threshold) can be generated.
    • Item Coverage: Percentage of items for which meaningful recommendations can be generated.

    Diversity Metrics¶

    • Intra-list Dissimilarity: Measures diversity within each user's recommendation list (1 − average pairwise cosine similarity between items).
    • Inter-diversity (Personalization): Measures how different recommendation lists are across users. Higher values indicate more personalized recommendations.

    Defensive Metric Computation

    All metrics are calculated in try...except blocks to ensure that a failure in one metric does not stop the entire experiment batch.

    5. Explainability Experiment¶

    In addition to the quantitative metric-based experiments, an explainability survey was conducted with 36 participants (19 with IT experience, 17 without).

    Methodology¶

    1. Scenarios were created for each algorithm–dataset combination, presenting: the algorithm name, feature values, a snippet of user history, the recommended item, and an explanation.
    2. Questions were divided into:
      • General satisfaction: Overall satisfaction with the explanation (1–5 scale).
      • Specific criteria based on the SAGES framework: Simple, Adaptable, Grounded, Extendable, Source-based (each rated 1–5).
    3. Explanation types varied by algorithm:
      • CBF: Item-to-item similarity-based explanation.
      • CF: User-to-user similarity-based explanation.
      • RL: Sequence-of-activity-based explanation (knowledge graph paths).

    6. Reinforcement Learning Pipeline¶

    The RL algorithm follows a distinct, multi-stage pipeline orchestrated by qpera/RL.py.

    1. Preprocessing: Extracts entities and relations from the raw data to build a knowledge graph structure.
    2. TransE Training: Trains a TransE model on the graph to learn embeddings for all entities and relations.
    3. Policy Training: Trains the Actor-Critic agent using PPO to learn how to navigate the graph.
    4. Inference: Uses a beam search algorithm to generate the top-k recommendation paths for each user in the test set.

    7. Caching & Performance¶

    To speed up development and repeated runs, the project uses two caching systems:

    • Dataset Cache: Processed and merged datasets are saved in qpera/datasets/<DatasetName>/. Subsequent loads read from this cache.
    • RL Artifact Cache: The RL pipeline saves its intermediate artifacts (knowledge graphs, embeddings, labels) to qpera/rl_tmp/<DatasetName>/.

    8. Troubleshooting¶

    MLflow Connection Issues¶

    • Problem: MLflow UI is not accessible or experiments are not logging.
    • Solution: 
      # Ensure the server is running
make run-mlflow

# If it's corrupted, force-clear the local tracking data
rm -rf mlruns/ mlflow.db

    RL Pipeline Fails¶

    • Problem: The RL experiment fails, often during training or graph creation.
    • Solution: The RL cache can sometimes become corrupted. Clear it for the specific dataset and retry. 
      # Example for MovieLens
rm -rf qpera/rl_tmp/MovieLens/

    Memory Errors¶

    • Problem: An experiment fails with an OutOfMemoryError.
    • Solution: Reduce the dataset size by editing the rows parameter in ExperimentRunner._define_experiments() in qpera/main.py. 
      {"algo": "CF", "module": CF, "func": "cf_experiment_loop", "dataset": "amazonsales", "rows": 5000},
